<a id="opt-20260314-1605-browser-preview"></a>
### 1、2026-03-14 16:05 ~ 16:30 浏览器 UI 预览支持

**背景**：开发时每次改 UI 都需要打 APK 安装到真机才能看效果，效率极低。
目标是让 `npm run dev` 后在浏览器里完整预览所有页面（含真实数据）。

#### 1、涉及文件及改动

| 文件 | 改动类型 | 具体内容 |
|------|----------|----------|
| `app/package.json` | 新增依赖 | `"sql.js": "1.14.1"`（WebAssembly 版 SQLite）|
| `app/package.json` | 新增开发依赖 | `"@types/sql.js"`（TypeScript 类型声明）|
| `app/vite.config.ts` | 修改 | 新增 `optimizeDeps: { include: ['sql.js'] }`，让 Vite 正确预构建 sql.js 的 CommonJS 模块 |
| `app/src/db/webDatabase.ts` | **新建文件** | 浏览器专用的 in-memory SQLite 实现，基于 sql.js（WebAssembly）|
| `app/src/db/database.ts` | 修改 | 新增 import、在 `init()` / `query()` / `run()` / `executeSet()` / `close()` 每个方法里加 `!Capacitor.isNativePlatform()` 判断，浏览器走 webDatabase，Android 走原有原生路径 |
| `app/src/composables/useCamera.ts` | 修改 | `takeGroupPhoto()` 浏览器返回 `'mock://browser-photo'`；`readPhoto()` 浏览器返回 `null` |

#### 2、`webDatabase.ts` 做了什么

```
1. 调用 initSqlJs() 加载 WebAssembly SQLite 引擎
2. 创建 in-memory 数据库（刷新页面数据会重置，这是正常行为）
3. 执行 CREATE_TABLES_SQL → SEED_DATA_SQL → TEST_SEED_SQL
   （和 Android 上首次安装时完全相同的 SQL，保证数据一致）
4. 导出 webQuery / webRun / webExecuteSet 三个函数供 database.ts 调用
```

#### 3、`database.ts` 的改动逻辑

```typescript
// 每个方法开头加的判断，其余代码完全不变
if (!Capacitor.isNativePlatform()) {
  // 走 sql.js 路径（浏览器）
  return webXxx(...)
}
// 以下走原来的 CapacitorSQLite 路径（Android）
```

#### 4、⚠️ 是否需要还原？

**结论：这批改动无需还原，可永久保留。** 原因如下：

- `isNativePlatform()` 在 Android 上始终为 `true`，浏览器分支代码对 Android **完全不生效**
- 原有所有 SQL 逻辑、仓库层、store 层**零改动**
- `tsc --noEmit` 零错误，`vite build` 零错误

#### 5、⚠️ 遗留问题（后期需处理）

**APK 体积增加**：`sql.js`（~700KB js + wasm）会被打包进 Android Assets 目录，
即使在 Android 上永远不会执行这些代码。

后期优化方案（暂不处理，等 UI 调试完成后再做）：

```typescript
// 方案A：在 database.ts 改为动态 import（只在浏览器时才加载 sql.js）
if (!Capacitor.isNativePlatform()) {
  const { initWebDatabase, webQuery } = await import('./webDatabase')
  // ...
}

// 方案B：vite.config.ts 用 build.rollupOptions.external 在 Capacitor 构建时排除 sql.js
// 方案C：接受现状（开发版 APK 多 700KB，生产版再优化）
```

## 七、遇到的问题和解决方案

### 1、ECharts 类型定义错误（2026-03-10 下午）

**错误信息**：
```
TS2339: Property 'ECharts' does not exist on type 'typeof import("echarts")'
```

**原因**：
ECharts v5 不再将 `ECharts` 作为 class 导出，不能用 `InstanceType<typeof echarts['ECharts']>`。

**解决方案**：
改用 `ReturnType<typeof import('echarts')['init']>` 推断实例类型。

```typescript
// 错误写法
let chart: InstanceType<typeof import('echarts')['ECharts']> | null = null

// 正确写法
let chart: ReturnType<typeof import('echarts')['init']> | null = null
```

**涉及文件**：`AttendanceStats.vue`、`BusinessStats.vue`

---

### 2、测试数据缺失（2026-03-10 下午）

**现象**：
初始 `SEED_DATA_SQL` 仅有 2 个用户账号 + 1 条预警阈值配置。安装后是空数据库，
所有功能无法正常测试（签到列表空、统计图空白、预警列表空）。

**解决方案**：
在 `schema.ts` 中新增 `TEST_SEED_SQL`，`database.ts` 的 `seedData()` 判断全新安装后一并写入：

- 15名学生（英语A班/数学B班/科学C班，覆盖红/黄/绿三种课时状态）
- 24条购课/续费记录（含金额，用于经营报表验证）
- 150条签到记录（10次课 Mon/Wed/Fri，到课/迟到/请假/旷课均有分布）
- 6条跟进记录（pending/contacted 两种状态）

**触发条件**：`seedData()` 检测 users 表 count=0 才写入，不影响已有数据。

---

### 3、attendances 表缺少唯一约束（2026-03-10 下午）

**现象**：
签到表没有 `UNIQUE(student_id, date)` 约束，可能对同一学生同一天重复插入，导致统计数据翻倍。

**解决方案**：
在 `CREATE TABLE attendances` 中增加 `UNIQUE(student_id, date)` 表级约束，
并移除了原来多余的 `idx_attendances_student_date` 普通索引（UNIQUE 约束自动建立等效索引）。

---

### 4、Agent 并发开发的文件同步问题（2026-03-10 上午）

**现象**：
多 Agent 并发开发时，backend-dev 已完成 `AttendanceToday.vue`（322行），
但 team lead 在检查时读到了临时状态，误判为未完成（只有9行占位符）。

**解决方案**：
不依赖 Agent 状态汇报，直接用 `wc -l` / `head` 命令验证文件实际行数和内容作为完成判据。

### 14、2026-03-14 18:27 ~ 18:33 全局手势导航：边缘左滑前进 / 右滑返回（含首页与末尾边界保护）

1. 作为移动端 App，页面导航缺少系统化手势操作；

  - 新增全局触摸手势监听（`touchstart/touchend/touchcancel`）；
  - 仅在边缘区域触发手势：
    - 左边缘右滑：返回（`router.back()`）；
    - 右边缘左滑：前进（`router.go(1)`）；
  - 增加阈值约束防误触：
    - 最小水平位移 `72px`
    - 最大垂直偏移 `64px`

    - 水平位移需显著大于垂直位移；

    - 首页（无 back）右滑不执行；

- 满足移动端基础导航体验；

**风险评估与边界**：
- 风险1：与页面横向滚动冲突  
  - 控制：仅边缘触发 + 位移/方向阈值双重过滤。
- 风险2：误触导致意外跳转  
  - 控制：短时大位移识别 + 交互控件区域屏蔽。
- 风险3：首页/末尾继续滑动报错  
  - 控制：先检查 `history.state` 的 `back/forward`，无可用历史时直接忽略，不执行导航。
- 边界外不改动：未修改任何业务页的路由逻辑、数据逻辑与组件结构，仅在 `App.vue` 增加导航手势层。
